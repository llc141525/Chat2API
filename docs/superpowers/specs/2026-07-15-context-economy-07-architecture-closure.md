# Context Economy 07: Architecture Closure And Legacy Path Retirement

## Objective

Close remaining bypasses after the first six implementation documents land.

This document exists because the architecture can still leak even if classifier, summary quality, digest, assembly, session boundary, and prompt budget are implemented locally. The remaining risks are old entry points, dedicated adapter paths, compatibility tests, and tool-history assumptions that can keep replaying large provider-visible context.

## Gap 1: OpenAI Session Identity Does Not Model All Boundaries

Current `OpenAISessionIdentity` in `src/main/proxy/routes/openaiSession.ts` only allows:

```ts
'normal' | 'client_compact' | 'tool_child' | 'subagent_child'
```

But lower layers use:

```ts
'normal' | 'client_compact' | 'server_summary' | 'summary_generator' | 'tool_child' | 'subagent_child'
```

### Required Change

Unify session boundary types across:

- `src/main/proxy/types.ts`
- `src/main/proxy/routes/openaiSession.ts`
- `src/main/proxy/sessionBoundary.ts`
- `src/main/proxy/promptBudgetPolicy.ts`
- `src/main/proxy/services/sessionBoundaryPlan.ts`

`openaiSession.ts` does not necessarily need to derive `server_summary` directly from client input, but its type model and diagnostics must understand all runtime boundary reasons.

### Tests

Add:

- `OpenAISessionIdentity type accepts runtime boundary reasons`
- `server_summary fork preserves parent identity and is visible in diagnostics`
- `summary_generator fork is never confused with client compact`

## Gap 2: Dedicated Provider Forwarders Bypass ProviderRuntime

`RequestForwarder` still has dedicated methods:

- `forwardGLM`
- `forwardQwen`
- `forwardKimi`
- `forwardMiniMax`
- `forwardMimo`
- `forwardPerplexity`
- etc.

ProviderRuntime pilot is gated by `CHAT2API_PROVIDER_RUNTIME_PILOT`. Currently default pilot covers Qwen and GLM, but not every plugin-backed provider.

### Required Change

Decide one explicit policy:

1. ProviderRuntime is the main path for all plugin-backed providers.
2. Dedicated forwarders are fallback-only and must call shared helpers for:
   - summary quality gate,
   - workflow digest,
   - session boundary plan,
   - prompt budget diagnostics.

Preferred target:

- make ProviderRuntime the default for every registered plugin;
- keep dedicated forwarders only behind an explicit emergency env flag;
- add a warning when dedicated path is used for a plugin-backed provider.

### Tests

Add:

- `registered provider plugins use ProviderRuntime by default`
- `dedicated fallback emits diagnostic`
- `dedicated GLM obeys SessionBoundaryPlan`
- `dedicated Qwen obeys SessionBoundaryPlan`

## Gap 3: Tool History Preservation Semantics Are Outdated

`ProviderToolProfile.preservesToolHistory` defaults to `true`, and many tests still assert that `## Available Tools` messages are preserved through context management.

This was useful to prevent tool-definition loss, but it now conflicts with the desired architecture:

- tool definitions must survive compact by re-derivation;
- old tool-contract messages must not be replayed as history.

### Required Change

Split the concept into two independent capabilities:

```ts
preservesToolExchangeHistory: boolean
requiresHistoricalToolContractReplay: boolean
```

Target default:

- `preservesToolExchangeHistory: true`
- `requiresHistoricalToolContractReplay: false`

Tool calls/results may need structured preservation. Tool contract text should be regenerated by `ToolCallingEngine`.

### Test Migration

Update tests that currently say:

```text
tool definition contract message must be preserved
## Available Tools message must be preserved
```

New expected behavior:

- historical `## Available Tools` may be stripped;
- current `ToolManifest.renderedPrompt` must still be present when tools are needed;
- catalog snapshot must be recoverable from session/current request/prompt-embedded fallback;
- assistant `tool_calls` and `tool_call_id` metadata must still be preserved.

Do not delete tests without replacing assertions.

## Gap 4: Prompt Budget Backward Compatibility Pins Old Behavior

Current tests include:

```text
undefined mode preserves current behavior (backward compatibility)
```

and assert that undefined mode includes system text, tool contract, user turn, and assistant response.

That can preserve the old full-prompt behavior indefinitely.

### Required Change

After `SessionBoundaryPlan` and diagnostics exist:

- make prompt refresh mode explicit for provider-runtime requests;
- treat undefined mode as a test-only legacy fallback;
- emit diagnostic when undefined mode reaches provider request assembly;
- eventually make undefined mode map to `digest` or `minimal` depending on boundary and catalog state.

### Tests

Add:

- `runtime requests always carry explicit promptRefreshMode`
- `undefined promptRefreshMode emits legacy diagnostic`
- `normal stable continuation uses minimal`
- `compact continuation uses digest unless tool_ready is required`

## Gap 5: Prompt-Embedded Tool Catalog Must Move To Config Lane

Prompt-embedded tool extraction (`prompt_embedded`) is currently a catalog source. That is correct, but the raw prompt block must not remain provider-visible history after extraction.

### Required Change

After extracting tools from prompt-embedded `## Available Tools`:

- store normalized catalog snapshot;
- strip raw prompt-embedded tool block from workflow history;
- regenerate the contract through `ToolManifest` if tools are needed.

### Tests

Add:

- `prompt_embedded catalog survives after raw tool block stripped`
- `prompt_embedded tools re-render through ToolManifest`
- `summary input excludes prompt_embedded raw block`

## Gap 6: Export Evidence Should Become Fixtures

The user's exported JSON revealed issues deterministic tests missed.

### Required Change

Create sanitized fixtures derived from:

- main 52-turn Qwen export shape;
- tool-call 103KB export shape;
- summary polluted export shape.

Do not commit sensitive raw exports. Commit reduced/redacted fixtures with the same structural markers and sizes scaled down where possible.

### Tests

Add:

- fixture replay produces bounded provider prompt;
- fixture replay summary quality rejects polluted summary input;
- fixture replay prompt diagnostics count repeated runtime/tool markers;
- fixture replay proves tool definitions re-derived after stripping historical contract.

## Gap 7: Raw Prompt Logging And Diagnostics Hygiene

Some current tests/logs print large prompt previews. As context economy diagnostics expand, logs must avoid raw prompt leakage.

### Required Change

Use structured diagnostics only:

- counts,
- byte sizes,
- marker counts,
- boundary action,
- session id source,
- prompt refresh mode.

Do not log full tool schemas, skill docs, or raw provider prompts except in explicit local debug mode.

### Tests

Add:

- diagnostics do not include raw skill document text;
- diagnostics do not include raw `## Available Tools` body;
- diagnostics include enough marker counts to debug regressions.

## Final Closure Checklist

- [ ] `OpenAISessionIdentity` and runtime boundary reason types are unified.
- [ ] ProviderRuntime is default for all registered plugins or dedicated fallbacks share the same plan.
- [ ] Tool-history tests assert re-derivation, not raw contract preservation.
- [ ] Undefined prompt-refresh compatibility mode is no longer used by runtime main path.
- [ ] Prompt-embedded catalogs are extracted into config lane and stripped from workflow history.
- [ ] Redacted export-shape fixtures exist and fail on repeated runtime/tool config replay.
- [ ] Diagnostics are structured and raw-prompt-safe.
